Music Playlist Redesign

Overview

Music Playlist Redesign is a project for a Usability and Information Architecture course done in a group of four members.

This project involved analyzing areas of improvement for playlist creation and management in Apple Music, Spotify, and SoundCloud and then creating mobile prototypes that better allow for playlist creation and management.

Design Problem

Creating playlists for a certain type of sound or through merging multiple playlists is a challenge for users of mainstream music applications (Spotify, Apple Music etc.) because these mainstream applications do not provide such features that users desire.

Design Process

Define Phase

The first step in the design process was identifying a general target audience, which we defined as users of mainstream music applications such as Spotify, Apple Music, or SoundCloud.

After identifying the target audience, the next step was to prepare for and conduct interviews. Nine interview questions were created in a cooperative manner by the entire team as a way to maintain consistency. These questions revolved around finding out in what contexts and how users create playlists on their preferred application. Each team member conducted interviews on 2-3 interviewees. The team then compiled the results and identified the top nine user scenarios that reflect cases when people create, manage, or use playlists.

The following step in the design process involved performing observations of users using two music applications to perform a task defined in a particular user scenario. Each team member performed observations for 2-3 of the user scenarios from the previous step and documented the workflow through screen recordings. After each scenario's observation, the better app and less good app in terms of workflow and user experience was noted down.

The team performed a competitive analysis of the information architecture among Spotify, Apple Music, and Spinrilla. This step involved creating graphs that show the hierarchy of the main pages on the application. This step also involved comparing how many genres are available to the user between the different apps.

Design Phase

In the design phase, the team started out with selecting two playlist app problems to solve through prototyping. The first problem revolved around allowing users to better create playlists for a particular type of sound. The second problem was related to allowing users to merge multiple playlists into one playlist.

After selecting which problems to work on, the team performed low-fidelity wireframing on a whiteboard to figure out the basic information architecture and functionality for both prototypes.

Based off of the low-fidelity wireframes, we moved onto prototyping using Figma to create high-fidelity representations of screens and used InVision to link the different screens together for both prototypes.

Lessons Learned

A main takeaway from this project was that user observations provide more meaningful information than interviews regarding which music application allows for a better workflow. This is since interviews only provide insight into opinions and user scenarios whereas observations build off of these user scenarios to show how well the user experience is for a particular music app.

I learned how to analyze information architecture in terms of how well it facilitates user interaction since, for example, SoundCloud gave users a hard to create a playlist due to how the playlist creation feature was hidden in the browsing songs feature.

A final important piece of knowledge gained was that defining the design problem as specific as possible and the minimum requirements for a viable solution go a long way in helping out the design process of creating the final prototype.

Obstacles Faced

During the competitive analysis between music applications, we struggled to identify which app provided a better workflow in certain scenarios. This was due to how Spotify and Apple Music both provided similar experiences for browsing new music and Spotify and SoundCloud both have the same mechanism to delete songs off of playlists. We overcame this obstacle by looking at the amount of screens the user had to go through and how easy the navigation to a particular feature was rather than the actual mechanism of the feature itself.

Another challenge the team faced was deciding which scenarios to prototype. Among the playlist creation scenarios, we were stuck between creating a playlist for a particular sound versus for a particular genre. We chose the scenario for a particular sound based off of how this scenario essentially refers to the one for a particular genre and even more for areas such as a particular mood or artist. The potential of this scenario to solve the problem of other scenarios led us to choose this one.

Another challenge was faced during prototyping the merging playlists scenario. We had trouble deciding if we should offer users the option to delete the existing playlists being merged together. This was because some group members suggested this feature would tax the cognitive workload of the user. However, this feature offers more flexibility to the user in completing the user scenario task, which outweighed the disadvantage of displaying too much to the user.

A final main obstacle faced during this project was not knowing what to include on the home screen of the prototypes. Since both prototypes revolved around a particular user scenario, the content on the home screen had to be carefully chosen so that it did not carry the user away from the user scenario. This obstacle was solved as we decided to put music recommendations on the home screen. This solution helped tackle the scenario of creating playlists for a particular type of sound and did not interfere with the merging playlists solution.
